# 经验积累

本文档用于持续积累项目推进中的经验。

## 当前已确认经验

### 经验 1：先建立最小闭环，再扩写教材

在当前阶段，先把 `MNIST` 跑通，比一开始全面铺开整本书更有效。

原因：

1. 有实际代码和结果，教材更容易写实
2. 更容易积累第一手问题和经验
3. 更容易形成后续章节的模板

### 经验 2：教材和代码必须同步设计

如果先写教材、不设计代码结构，后续文档和代码很容易脱节。

### 经验 3：需要单独的 meta 目录

很多真正有价值的方法论内容，不属于教材正文，也不属于代码实现，需要单独沉淀。

### 经验 4：教学代码应按阶段递进，而不是一开始就工程化

背景：

在为 `MNIST` 设计代码目录时，提前搭出完整工程骨架会让初学者在真正理解任务之前先被目录复杂度吓到。

结论：

教学型代码更适合按阶段递进组织，让代码结构随着理解一起生长。

为什么重要：

1. 初学者更容易先建立最小闭环
2. 学习者更容易理解“为什么需要拆分和抽象”
3. 工程化结构可以后置出现，而不是开局抢占注意力

适用范围：

适用于本项目当前的 `MNIST` 阶段，也适用于后续 `CNN`、`ResNet`、`Transformer`、`Diffusion` 的教学型代码组织。

是否值得抽象成 skill：

值得，属于“教学型项目结构设计”中的稳定方法。

### 经验 5：共享目录应优先抽取“数据”，而不是过早抽取训练框架

背景：

在教学阶段，过早抽取 `common`、`engine`、`metrics` 这类训练框架容易增加理解负担，但数据目录如果不共享，又会导致每个阶段重复下载同一份数据。

结论：

应优先抽取 `code/data/` 这类低认知负担、强复用价值的公共目录。

为什么重要：

1. 它减少重复而不会显著增加复杂度
2. 它适合初学者理解
3. 它能为后续多个阶段和多个数据集复用打基础

适用范围：

适用于当前 `MNIST` 阶段，也适用于后续 `Fashion-MNIST`、`CIFAR-10` 等数据集。

是否值得抽象成 skill：

值得，属于“教学型代码仓中哪些公共目录应先抽取”的稳定经验。

### 经验 6：教材应追求高信息密度，而不是堆字数

背景：

如果教材写得过长，学习者会更难快速抓住当前阶段真正需要执行的动作。

结论：

教材应尽量压缩冗余背景，优先保留关键结论、命令、代码和必要解释。

为什么重要：

1. 更适合边学边做
2. 更容易形成清晰的执行路径
3. 更符合本项目“book 指导 code”的工作方式

适用范围：

适用于本项目当前和后续所有教材章节。

是否值得抽象成 skill：

值得，属于“教学型内容写作约束”中的稳定经验。

### 经验 7：教材里能指导重新下载、重新生成的东西，默认都不该进 git

背景：

像数据集、模型权重、日志和输出图片这类内容，通常体积较大，而且可以通过教材中的步骤重新获取或重新生成。

结论：

只要教材已经能指导用户重新下载或重新生成，这类内容默认不应进入版本库。

为什么重要：

1. 可以保持仓库干净
2. 可以降低版本库体积
3. 可以减少无意义的二进制或产物变更

适用范围：

适用于数据集、输出目录、权重文件、日志文件和其他运行产物。

是否值得抽象成 skill：

值得，属于“教学型项目仓库边界管理”中的稳定经验。

### 经验 8：教材中凡是要求用户操作的地方，都必须给出明确可执行指令

背景：

如果教材只写“用某工具安装”或“用某方式下载”，新手通常并不知道具体应该敲什么命令。

结论：

教材里只要涉及用户操作，就应尽量给出可以直接执行的命令、路径和文件名。

为什么重要：

1. 能显著降低新手的停顿点
2. 能把教材从“解释型内容”提升为“可执行指导”
3. 更符合本项目“book 指导 code”的工作方式

适用范围：

适用于安装环境、下载数据、运行脚本、创建目录、验证结果等所有操作型内容。

是否值得抽象成 skill：

值得，属于“教学型写作可执行性约束”中的稳定经验。

### 经验 8：环境验证不能只看 import 和版本号，还要验证关键库之间的互操作

背景：

在教学环境里，如果安装命令只写成“直接装最新版”，学习者可能会得到一个“能 import、能打印版本号，但关键功能并不真的可用”的依赖组合。

例如 `torch` 与 `numpy` 存在二进制兼容问题时，`import torch` 可能仍然成功，但 `torch.from_numpy(...)` 或 `tensor.numpy()` 会直接失败。

结论：

教材中的环境验证，不应只停留在“能 import”或“能打印版本号”，而应至少覆盖当前阶段真实会用到的一次关键互操作。

为什么重要：

1. 可以更早暴露“假通过”的环境问题
2. 可以减少学习者把环境问题误判成代码问题
3. 可以让教材步骤更接近真正可运行、可复现的状态

适用范围：

适用于本项目所有涉及第三方数值库、深度学习框架、可视化库或二进制依赖的章节。

是否值得抽象成 skill：

值得，属于“教学型项目中的环境检查与可复现性设计”经验。
