# 项目原则

## 1. 项目不是单纯练代码

本项目的目标不是只把模型跑通，而是同时产出：

1. 可运行代码
2. 教材内容
3. 方法论沉淀
4. 可复用 skill 雏形

## 2. 教材与代码必须联动

1. 重要教材章节应有对应代码实现
2. 重要代码实现应能回指教材章节
3. 文档和代码不能长期脱节

## 3. 先闭环，再扩展

1. 优先建立最小可运行版本
2. 优先形成当前阶段完整闭环
3. 再进入更复杂主题

## 4. 先实践，再总结

1. 先把问题做出来
2. 再把过程讲清楚
3. 再把方法抽象出来

## 5. 每次推进都要考虑沉淀价值

每次新增代码、文档或计划时，都尽量问自己三个问题：

1. 这是否帮助当前学习
2. 这是否有助于教材写作
3. 这是否值得沉淀到 `meta/`

## 6. 教材写作要高信息密度

教材写作应尽量少堆字数。

要求是：

1. 用尽量少的字表达尽量明确的动作和判断
2. 少讲空泛背景，多讲当前阶段真正需要的信息
3. 少写重复解释，多给可执行命令、关键结论和必要说明

衡量标准不是“写了多少”，而是“学习者看完后能不能立刻往下做”。

## 7. 教材规范应独立沉淀并持续完善

教材写作不只是临时措辞问题，而应有独立原则持续维护。

要求是：

1. 稳定的教材写作约束应优先写入 `meta/01-原则/教材编写原则.md`
2. 后续发现新的教材写作问题时，应优先补充规范，而不是只做一次性修改
3. 教材正文、教材经验和教材原则三层内容应尽量分开

这样做的目的，是让教材质量随着项目推进逐步稳定，而不是依赖单次记忆。

## 8. 可重新下载或重新生成的内容默认不进 git

如果某类内容满足以下条件之一：

1. 教材已经提供了明确下载方式
2. 教材已经提供了明确生成方式
3. 它本身属于运行产物、缓存或中间结果

那么这类内容默认不应提交到 git。

典型例子包括：

1. 下载型数据集
2. 模型权重
3. 日志文件
4. 输出图片
5. 中间缓存

只有说明文档、目录骨架和真正需要版本管理的源码才应保留在仓库中。

## 9. 教材中凡是要求用户操作的地方，都必须给出明确可执行指令

如果教材要求用户安装、下载、运行、创建、修改或验证某件事，就不能只写“使用某工具”或“通过某方式”。

必须尽量给出：

1. 具体命令
2. 具体路径
3. 具体文件名
4. 预期结果

判断标准很简单：

学习者看完后，应该能直接在终端里执行，而不是自己猜下一步怎么做。
