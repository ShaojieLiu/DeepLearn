# 教材评审规则

本文档用于为本项目中的教材内容建立相对稳定的评审维度、评分标准和比较口径。

它主要服务于主编、编辑、审稿人和后续 agent 的持续评审，而不是直接面向读者。

目标不是把评审做成形式化打分游戏，而是让后续的修改、复审和版本对比有一致标准。

## 1. 评审使用原则

评审时优先关注以下问题：

1. 这篇内容能不能真正帮助读者往前做
2. 这篇内容有没有把关键边界讲准
3. 这篇内容和代码、命令、结果有没有对应起来
4. 这篇内容在整套教材里承担的职责是否清楚
5. 这篇内容还有哪些最值得优先修改的短板

评审的目标不是平均用力，而是优先暴露最影响学习效果的问题。

## 2. 评分方式

每个维度统一采用 `0` 到 `5` 分。

评分含义如下：

### 0 分

严重缺失。

这一维度几乎没有建立，或者存在明显误导，已经影响读者正常学习。

### 1 分

明显不足。

已经涉及这一维度，但内容很弱，读者很难据此获得稳定帮助。

### 2 分

基础存在，但质量偏低。

能看出作者想解决这个问题，但解释、结构或证据明显不够。

### 3 分

达到可用线。

读者基本能继续往下学，但仍有不少可以明显优化的地方。

### 4 分

质量较好。

这一维度已经较稳定，只有局部细节需要打磨。

### 5 分

质量优秀。

这一维度不仅没有明显短板，而且足够清晰、稳健、可复用，能够作为后续章节的参考样板。

## 3. 核心评审维度

建议默认按以下八个维度评审。

### 维度 1：范围与完成度匹配

看教材对外承诺的范围，是否和当前真实完成度一致。

重点检查：

1. 目录规模是否明显超前于正文完成度
2. 当前章节是否被包装成“已成熟”，但实际上只是草稿
3. 是否清楚区分了已完成内容和后续规划

评分参考：

1. `0-1` 分：承诺严重超前，读者容易被误导
2. `2` 分：部分内容已完成，但整体完成度表达模糊
3. `3` 分：基本匹配，但局部仍有过度承诺
4. `4-5` 分：范围收敛清楚，完成度表达真实

### 维度 2：结构推进与章节职责

看这一章在整套教材中的位置是否明确，是否真的承担了自己该承担的职责。

重点检查：

1. 这一章是在讲导读、概念、实操还是复盘
2. 章节内部是否围绕同一个主任务推进
3. 是否和前后章节产生大段职责重叠

评分参考：

1. `0-1` 分：章节职责混乱，读者不知道这章到底要解决什么
2. `2` 分：主任务存在，但重复和偏题较多
3. `3` 分：基本清楚，但还能继续压缩和聚焦
4. `4-5` 分：章节边界清晰，推进自然

### 维度 3：技术准确性与边界清晰度

看技术表述是否准确，尤其是容易混淆的关键边界有没有讲清。

重点检查：

1. 概念定义有没有偷换
2. 术语、代码行为、命令效果是否一致
3. 是否把容易混淆的相近概念边界讲清
4. 是否把不同阶段、不同条件下的行为差异区分清楚

必要时可以用具体例子辅助说明，但例子只是辅助判断方式，不应被理解为本维度的完整范围。

评分参考：

1. `0-1` 分：存在明显技术错误或误导
2. `2` 分：整体方向对，但关键边界模糊
3. `3` 分：基本准确，个别表述仍需修正
4. `4-5` 分：准确且边界清晰，读者不易被带偏

### 维度 4：可执行性与证据链完整度

看讲解是否能落到真实操作，读者能不能照着做，并判断自己是否做对。

重点检查：

1. 是否有对应代码、脚本或目录
2. 是否给出真实命令
3. 是否说明预期输出或预期现象
4. 是否能据此定位失败点

评分参考：

1. `0-1` 分：只有讲解，没有可验证落点
2. `2` 分：有部分命令或代码，但证据链不完整
3. `3` 分：可以执行，但验证信息还不够充分
4. `4-5` 分：代码、命令、结果和判断标准形成闭环

### 维度 5：解释落地性与可跟踪性

看抽象概念有没有被落到具体样本、具体形状、具体流转过程上。

重点检查：

1. 是否有输入输出的具体例子
2. 是否解释了关键对象或中间状态的变化过程
3. 是否说明了为什么这样设计
4. 是否提示了常见错法和典型现象

评分参考：

1. `0-1` 分：主要停留在抽象概念层
2. `2` 分：给了例子，但仍然难以跟踪理解
3. `3` 分：能帮助基本理解，但中间桥接还不够
4. `4-5` 分：概念、代码和现象能互相对应

### 维度 6：信息密度与冗余控制

看内容是否紧凑，是否存在明显重复、绕行或低信息密度段落。

重点检查：

1. 是否反复重复同一个总原则
2. 是否有一大段背景，但对当前动作帮助很小
3. 是否存在删掉后不影响主线推进的内容

评分参考：

1. `0-1` 分：重复严重，读者容易失去耐心
2. `2` 分：有不少可压缩内容
3. `3` 分：基本紧凑，但局部仍有重复
4. `4-5` 分：信息密度高，推进清楚，不拖泥带水

### 维度 7：读者行动指引清晰度

看读者读完以后，是否知道下一步该做什么。

重点检查：

1. 本章目标是否明确
2. 读者现在该执行什么动作是否明确
3. 本章结束后，读者是否知道该进入哪一步

评分参考：

1. `0-1` 分：读完后不知道该怎么继续
2. `2` 分：方向大概知道，但动作不够清楚
3. `3` 分：能够继续推进，但指导还可更明确
4. `4-5` 分：行动路径清晰，读者容易顺着做下去

### 维度 8：教材与代码联动度

看教材和 `code/` 是否真正构成双线联动，而不是各说各话。

重点检查：

1. 教材提到的目录和脚本是否真实存在
2. 代码结构变化是否能回到教材中解释原因
3. 教材是否能帮助读者理解代码为什么这样组织

评分参考：

1. `0-1` 分：教材和代码明显脱节
2. `2` 分：有对应关系，但联系较弱
3. `3` 分：基本联动，但还不够稳定
4. `4-5` 分：教材和代码互相支撑，读者能顺着两条线学习

## 4. 总分解释

默认总分为 `40` 分。

建议按以下区间理解：

1. `0-15` 分：当前不适合直接给读者使用，需要重写或重构
2. `16-24` 分：可作为内部草稿继续推进，但不适合作为稳定教材内容
3. `25-31` 分：达到可发布的基础线，但仍有明显短板
4. `32-36` 分：整体较成熟，可作为当前阶段的合格章节
5. `37-40` 分：质量优秀，可作为后续章节的样板

总分只用于快速比较，不替代具体问题判断。

如果某个关键维度过低，即使总分不差，也应优先返工。

尤其是以下维度，不建议低于 `3` 分：

1. 技术准确性与边界清晰度
2. 可执行性与证据链完整度
3. 读者行动指引清晰度

## 5. 评审输出格式建议

每次评审建议至少输出以下内容：

1. 本次评审对象
2. 八个维度的单项评分
3. 总分
4. 最严重的三个问题
5. 最值得优先修改的三个动作
6. 本轮修改后希望重点复查的维度

推荐格式如下：

```text
评审对象：

单项评分：
1. 范围与完成度匹配：4/5
2. 结构推进与章节职责：3/5
3. 技术准确性与边界清晰度：2/5
4. 可执行性与证据链完整度：3/5
5. 解释落地性与可跟踪性：2/5
6. 信息密度与冗余控制：3/5
7. 读者行动指引清晰度：4/5
8. 教材与代码联动度：2/5

总分：23/40

主要问题：
1. ...
2. ...
3. ...

优先修改建议：
1. ...
2. ...
3. ...

下轮复查重点：
1. ...
2. ...
```

## 6. 使用约束

评审时应注意：

1. 评分要尽量基于正文、代码和可验证材料，不只基于写作意图
2. 先指出最影响读者学习的问题，再谈风格偏好
3. 同一套标准应尽量在不同章节上重复使用，方便横向对比
4. 当评审中出现新的稳定判断标准时，应回写到本文件，而不是只停留在单次意见里

## 7. 建议项：练习、检查点与反馈机制

这一项默认不纳入八个核心维度总分，但主编评审时建议单独检查。

原因：

1. 它对教材成书质量很重要
2. 但在项目早期，部分章节可能还处在主线搭建阶段
3. 先将其作为建议项，更适合逐步补齐，而不是一开始就压成硬门槛

建议检查：

1. 本章是否有简短的自检问题，帮助读者判断自己是否真的理解
2. 本章是否有低成本动手练习，而不只是连续阅读
3. 本章是否给出做对、做错时的反馈线索
4. 本章是否帮助读者形成“我现在会了什么、下一步该练什么”的闭环

可使用三档标记：

1. `缺失`：没有练习、检查点或反馈设计
2. `基础`：已有其中一部分，但不成体系
3. `完善`：练习、检查点和反馈机制基本形成闭环
